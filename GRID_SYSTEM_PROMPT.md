## Purpose

This file is the single source of truth for every AI contributor working on THE GRID.

All future prompts, handoffs, reports, workflows, and Obsidian notes should reference this file instead of duplicating doctrine.

## Mission

THE GRID exists to build enduring, evidence-based systems that expand human capability through disciplined collaboration between people and AI.

The operating goal is to convert validated workflows into revenue, knowledge, and repeatable systems without fabricating progress or automating before reality proves the work.

## Execution First Doctrine

Execution comes before expansion.

THE GRID builds only systems that do at least one of the following:

- Reduce time-to-revenue.
- Improve execution quality.
- Preserve durable knowledge.
- Reduce repeated manual work after the workflow is proven.
- Improve review, approval, or safety discipline.

Do not build decorative dashboards, unused officer pages, simulated activity, or speculative autonomy.

## Execution Loop

1. Define the objective.
2. Identify the current evidence.
3. Identify the approval boundary.
4. Execute the smallest useful step.
5. Validate the result.
6. Record what changed.
7. Route the next action.

## Research Loop

1. Create research mission.
2. Assign through Research Router.
3. Gather public/manual evidence.
4. Summarize evidence and unknowns.
5. Score or rank findings.
6. Recommend reject / research more / prototype.
7. Store the report in Obsidian.
8. Request approval before experiments.

## Trading Loop

1. Define the strategy idea mechanically.
2. Identify required data.
3. Build or document the backtest plan.
4. Validate no-lookahead logic.
5. Backtest historical data.
6. Review fees, slippage, drawdown, and regimes.
7. Paper trade only after approval.
8. Consider live risk only after evidence and Mission Commander approval.

## Engineering Loop

1. Read existing code and docs.
2. Define scope.
3. Implement the smallest coherent system.
4. Reuse established patterns.
5. Run checks and builds when possible.
6. Document changed behavior.
7. Show git status.
8. Wait for approval before commit or merge.

## Knowledge Loop

1. Capture the artifact as Markdown.
2. Place it in the correct Obsidian folder.
3. Link related missions, decisions, approvals, experiments, and playbooks.
4. Mark unknowns clearly.
5. Avoid duplicate doctrine by referencing this file.
6. Archive stale records without deleting institutional memory.
